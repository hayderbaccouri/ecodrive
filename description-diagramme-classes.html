EcoDrive

Description du diagramme de classes
Modèle de données — Base MySQL
Projet de fin d'études • Juin 2026

1. Présentation

Le diagramme de classes ci-dessous modélise la structure persistante de la plateforme EcoDrive. Il représente les quatre entités stockées en base de données MySQL, leurs attributs, leurs clés et les relations qui les lient. Le projet étant développé en PHP procédural (sans framework ORM), ce diagramme correspond directement au schéma relationnel implémenté via les instructions CREATE TABLE.

2. Les quatre entités

2.1 utilisateur

Stocke les comptes de la plateforme avec deux rôles possibles : client (visiteur inscrit) et admin (gestionnaire). Le mot de passe est haché avec bcrypt. L'email est unique pour éviter les doublons. Des champs optionnels (reset_token, reset_expires) gèrent la réinitialisation du mot de passe avec un lien à durée limitée (1 heure).

2.2 voiture

Catalogue des véhicules électriques proposés. Chaque voiture est identifiée par sa marque, son modèle, son année et son prix. Les données détaillées (spécifications techniques : puissance moteur, capacité batterie, autonomie) sont stockées côté PHP dans un tableau associatif, et non en base — ce choix a été fait pour éviter de sur-normaliser des données rarement modifiées.

2.3 réservation

Entité association qui matérialise la relation many-to-many entre utilisateur et voiture. Elle porte les attributs de la réservation d'essai : date, créneaux horaires (heure_debut, heure_fin), statut (pending / confirmed / cancelled), et notes optionnelles.

Un trigramme de validation empêche le double-book : pour une même voiture, un même jour, et des créneaux qui se chevauchent, la réservation est refusée si une réservation non-annulée existe déjà.

2.4 borne

Catalogue des bornes de recharge Exicom. Cette table est indépendante (sans clé étrangère) — elle n'est liée à aucune réservation dans la version actuelle. Elle référence 4 modèles de 3 à 22 kW.

3. Relations

RelationTypeDescription
utilisateur → réservation 1...* Un utilisateur peut effectuer plusieurs réservations d'essai
voiture → réservation 1...* Une voiture peut être réservée plusieurs fois (à des créneaux différents)
utilisateur → voiture (via réservation) *...* Relation many-to-many : un utilisateur peut essayer plusieurs voitures, une voiture peut être essayée par plusieurs utilisateurs

Ces relations sont implémentées physiquement par deux clés étrangères dans la table reservation :

  • fk_reservation_utilisateur : utilisateur_id → utilisateur(id_utilisateur) ON DELETE CASCADE
  • fk_reservation_voiture : voiture_id → voiture(id_voiture) ON DELETE CASCADE

Le CASCADE garantit que si un utilisateur ou une voiture est supprimé, les réservations associées le sont automatiquement — maintenant l'intégrité référentielle sans action manuelle.

4. Contraintes d'intégrité

  • Domaine : les champs role (utilisateur) et statut (réservation) sont des ENUM — les valeurs sont contraintes à un ensemble défini.
  • Référentielle : les clés étrangères avec CASCADE préservent la cohérence des données liées.
  • Unicité : l'email est unique — pas de doublon de compte.
  • Temporelle : la requête de détection de conflit horaire empêche les chevauchements de réservation (même voiture, même jour, même plage horaire).

5. Moteur de stockage

Toutes les tables utilisent le moteur InnoDB avec le jeu de caractères utf8mb4_unicode_ci. InnoDB est choisi pour sa gestion des transactions, l'intégrité référentielle (clés étrangères) et sa robustesse en production.

6. Synthèse

Le diagramme de classes reflète un schéma relationnel simple, cohérent et fonctionnel : 4 tables, des clés étrangères assurant l'intégrité, et des contraintes ENUM limitant les valeurs autorisées. L'absence de normalisation excessive dans voiture (les specs sont en PHP) constitue un choix pragmatique adapté à un catalogue de taille modeste (14 entrées). La table borne, bien qu'indépendante aujourd'hui, pourrait être reliée à future version via une table de réservation de borne.
